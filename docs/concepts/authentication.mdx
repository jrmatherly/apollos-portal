Apollos AI uses Microsoft Entra ID for authentication across all three components (portal, API, CLI). Each component uses a different OAuth 2.0 flow optimized for its context.

## Authentication flows

<Tabs>
<Tab title="Portal (frontend)">
  The portal uses **PKCE authorization code flow** via `@azure/msal-browser` v5.

  1. User clicks **Sign in with Microsoft**
  2. Browser redirects to Microsoft's authorization endpoint
  3. User authenticates and consents
  4. Browser receives an authorization code
  5. MSAL exchanges the code for access and ID tokens
  6. The portal sends the access token to the backend API with each request
</Tab>

<Tab title="API (backend)">
  The backend uses **client credentials flow** for service-to-service calls (Microsoft Graph, LiteLLM) and validates user access tokens from the frontend.

  - User tokens are validated by calling **Microsoft Graph `/me`** with the bearer token — if Graph accepts it, the token is valid
  - The `oid` (object ID) claim from the Graph response identifies the user
  - The JWT `aud` (audience) claim is verified against the app registration's client ID
  - The JWT `roles` claim is extracted for admin role checks (e.g., `portal_admin`)
  - Security group memberships are resolved via Microsoft Graph client credentials flow
</Tab>

<Tab title="CLI">
  The CLI uses **device code flow** via the MSAL Python library.

  1. User runs `apollos login`
  2. CLI displays a URL and a code
  3. User opens the URL in a browser and enters the code
  4. After authentication, the CLI receives tokens
  5. Tokens are cached locally for subsequent commands
</Tab>
</Tabs>

## Team provisioning

When a user first authenticates, the backend provisions them by:

1. Validating their Entra ID access token
2. Querying Microsoft Graph for their security group memberships
3. Matching group IDs against the `teams.yaml` configuration
4. Creating or updating the user's LiteLLM team memberships
5. Returning the user's profile and team list

This happens automatically on the `/api/v1/provision` endpoint, called after the first successful login.

## Token validation

The backend validates every request by:

1. Checking the `Authorization: Bearer <token>` header
2. Calling Microsoft Graph `/me` with the token — a `401` from Graph means the token is invalid or expired
3. Verifying the JWT `aud` claim matches the configured `AZURE_CLIENT_ID`
4. Extracting the user's `oid`, name, and email from the Graph response
5. Extracting the `roles` claim from the JWT payload for admin role checks

<Note>
  The backend uses the `HTTPBearer` security scheme. Missing or invalid tokens return `401 Unauthorized`. If Graph API is unreachable (non-401 error), the backend returns `503 Service Unavailable` instead of failing silently.
</Note>

## Entra ID setup

The portal requires an App Registration in Microsoft Entra ID with specific permissions and security group configuration. See the [Entra ID setup guide](/administration/entra-id-setup) for step-by-step instructions.
