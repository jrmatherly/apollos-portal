Teams are the core organizational unit in Apollos AI. Each team maps to both a Microsoft Entra ID security group and a LiteLLM team, connecting identity management with LLM access controls.

## How teams work

A team defines:

- **Model access** — which LLM models team members can use (e.g., `gpt-4o`, `claude-sonnet-4-20250514`)
- **Budget limits** — maximum spend per team over a billing period
- **Rate limits** — requests per minute and tokens per minute caps
- **Member list** — users whose Entra ID security group memberships include this team

## Team configuration

Teams are defined in `teams.yaml` (gitignored, use `teams.yaml.example` as a template). Each entry maps an Entra ID security group to a LiteLLM team:

```yaml
teams:
  - entra_group_id: "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
    team_alias: "Engineering AI Team"
    models:
      - "gpt-4o"
      - "gpt-4o-mini"
      - "claude-sonnet-4-20250514"
    max_budget: 500
    budget_duration: "30d"
    team_member_budget: 25
    litellm_role: "user"

  - entra_group_id: "yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy"
    team_alias: "All AI Users"
    models: ["gpt-4o-mini"]
    max_budget: 50
    budget_duration: "30d"
    is_default: true
```

### Configuration reference

| Field                | Required | Default  | Description                                                                         |
| -------------------- | -------- | -------- | ----------------------------------------------------------------------------------- |
| `entra_group_id`     | Yes      | —        | Object ID of the Entra ID security group                                            |
| `team_alias`         | Yes      | —        | Display name for the team in LiteLLM and the portal                                 |
| `models`             | No       | `[]`     | LLM models this team can access                                                     |
| `max_budget`         | No       | `0`      | Maximum spend for the team per budget period (0 = unlimited)                        |
| `budget_duration`    | No       | `"30d"`  | Budget reset period (e.g., `"30d"`, `"7d"`)                                         |
| `team_member_budget` | No       | `0`      | Per-user budget cap within this team (0 = unlimited)                                |
| `litellm_role`       | No       | `"user"` | LiteLLM team membership role (`"user"` or `"admin"`)                                |
| `is_default`         | No       | `false`  | Mark as a fallback team — suppressed when the user qualifies for a non-default team |

### Team member roles

The `litellm_role` field controls the user's role within the LiteLLM team:

| Role    | Description                                                         |
| ------- | ------------------------------------------------------------------- |
| `user`  | Standard team member — can use assigned models within budget limits |
| `admin` | Team administrator — can manage team members and settings           |

### Default (fallback) teams

A team with `is_default: true` acts as a fallback. During provisioning, if a user qualifies for at least one non-default team, default teams are suppressed — the user is only provisioned into their non-default teams. Default teams are only used when they are the user's sole option.

This prevents users in a specific team (e.g., "Engineering AI Team") from also being provisioned into a generic "All AI Users" team, which would create an unnecessary extra API key and clutter the Teams page.

<Warning>
  `teams.yaml` contains Entra ID group IDs and is gitignored. Never commit this
  file. Use `teams.yaml.example` as a starting template.
</Warning>

### Gate groups

The optional `required_groups` field controls who can access the portal, independent of team assignment:

```yaml
required_groups:
  - "<Object ID of SG-AI-Platform-Users>"
```

If `required_groups` is set, a user must be a member of at least one listed group to sign in. If omitted, membership in any team group is sufficient. This lets you add someone to the gate group before assigning them to a team, or remove portal access without touching team groups.

## Provisioning flow

When a user signs in for the first time, provisioning auto-triggers and the portal shows a progress dialog. The backend:

1. Resolves the user's Entra ID security group memberships via Microsoft Graph
2. Matches those groups against the `teams.yaml` configuration
3. Filters out default teams if the user qualifies for any non-default team
4. Creates or updates LiteLLM team memberships for the remaining matched groups
5. Generates an initial API key for each provisioned team
6. Returns the user's teams, model access, budget information, and generated keys

Users are automatically added to or removed from teams based on their current security group memberships. Changes in Entra ID are reflected the next time the user signs in.

## Viewing teams

In the portal, go to **Teams** in the sidebar to see:

- Teams you belong to with an **Active** status badge
- Team member count — click to see the full member list with roles
- Available models per team
- Current budget usage and limits with a visual progress bar
