---
title: "API keys"
description: "Create, manage, and use API keys for LLM proxy access"
---

API keys authenticate your requests to the LiteLLM proxy. Each key is scoped to a specific team and inherits that team's model access, rate limits, and budget.

## Creating keys

<Steps>
<Step title="Open API Keys">
  In the portal, go to **API Keys** in the sidebar.
</Step>
<Step title="Create a new key">
  Click **Create key**, provide a name, and select the team to associate it with.
</Step>
<Step title="Copy the key">
  Copy the generated key immediately. It is displayed only once and cannot be retrieved later.
</Step>
</Steps>

You can also create keys via the API:

```bash
curl -X POST /api/v1/keys/new \
  -H "Authorization: Bearer <access-token>" \
  -H "Content-Type: application/json" \
  -d '{"team_id": "engineering"}'
```

## Using keys

Include your API key in the `Authorization` header when making requests to the LiteLLM proxy:

```bash
curl -X POST https://your-litellm-proxy/chat/completions \
  -H "Authorization: Bearer sk-your-api-key" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role": "user", "content": "Hello"}]
  }'
```

The proxy validates the key, checks team permissions for the requested model, and routes the request to the appropriate LLM provider.

## Key naming

Keys are automatically named using the format `{email-prefix}-{team-name}` (e.g., `jsmith-engineering-ai-team`). If a collision occurs with an existing active key, a random suffix is appended (e.g., `jsmith-engineering-ai-team-a1b2c3d4`).

## Key status

Each key has one of the following statuses:

| Status | Description |
|--------|-------------|
| `active` | Key is valid and can be used for API requests |
| `expiring_soon` | Key expires within 14 days — consider rotating |
| `expired` | Key has passed its expiration date and is no longer valid |
| `rotated` | Key was replaced by a newer key via rotation |
| `revoked` | Key was manually revoked and permanently blocked |

Keys expire automatically based on the `default_key_duration_days` setting configured for your account.

## Managing keys

From the **API Keys** page you can:

- **View** all your keys with their status, creation date, expiration, spend, and associated team
- **Rotate** a key to expire the old key and generate a new one linked to it
- **Revoke** keys that are no longer needed

### Rotation vs. revocation

These operations differ in how the old key is handled:

| | Rotation | Revocation |
|---|---|---|
| **Old key** | Expired (deleted from LiteLLM) | Blocked (soft-deleted, preserves spend history) |
| **New key** | Created with `rotated_from_id` linking to old key | No new key created |
| **Use case** | Scheduled credential refresh | Compromised or unused key |

<Warning>
  Both operations are irreversible. Rotation creates a replacement key automatically; revocation does not.
</Warning>

## Key scoping

Each key is scoped to exactly one team. If you belong to multiple teams, create separate keys for each. This lets you:

- Track usage per team independently
- Rotate keys for one team without affecting others
- Match keys to specific applications or environments
